# DESIGN — make-account-item（レシート版デモ）

本番システム `make-account-item`（実在の会計事務所で 2025 年に本番デプロイ・日常使用）の
**レシート経路**を、合成データと自分の匿名レシートだけで**動く形に抽出**したデモの設計ドキュメント。

各記述には実態を明示するタグを付ける：

- 【本番✓】… 本番コードで実証済み（`make-account-item` の `improve-receipt-extract-text-performance` ブランチ）
- 【デモ】… このデモで再現する／簡略化する（実装方針。未実装の場合あり）

---

## 1. 何を・なぜ作ったか（顧客課題）

事務所の証憑データ化には **レシート / クレジットカード明細 / 銀行通帳** の 3 種があり、
**レシートを最優先**で自動化した。理由は次の 3 点：【本番✓】

1. **量が多い** — レシートは枚数が圧倒的に多い。
2. **外注コストが高く、実際に払っていた** — 既存の証憑データ化代行 **STREAMED に 1 枚 ≒ ¥20 を実際に支払っていた**。
   これを内製で置き換えれば、料金は Gemini API コストのみ＝**1 枚あたり ¥1 未満**（後述・デモで実測）。
3. **クレカ明細・銀行通帳は痛みが小さい** — カード会社や会計ソフト側に **API 連携機能が既にあり**、
   そもそも半ば構造化済みで優先度が低い。

加えてレシートは **3 種の中で最難関**だった（非定型：店ごとにレイアウトが違う・複数明細・税率 10%/8% 混在・
紙のかすれ）。そこで「**一番難しいレシートで型を確立し、後からクレカ・通帳へ横展開する**」という方針を取った。
このデモが見せるのは、その**確立した型（agent コア）**そのものである。

## 2. アーキテクチャ（デモ）

本番と同じ **2 段構成**（本番 `extract_text` → per-page `infer`）。1 PDF に複数レシートが入るため、
まず抽出してレシート配列にし、**レシートごとに**推論エージェントへ流す（＝本番の「ページ単位 infer」相当）。
実装中に「抽出＋推論を 1 回のマルチモーダル呼び出しに集約」を試したが、6 枚を 1 セッションで処理すると
最後の `validate-infer` 呼び出しが `MALFORMED_FUNCTION_CALL` で破綻した。本番どおり 2 段に分け、推論段の
入力をレシート 1 枚分の抽出済テキストにすると安定した（＝本番設計の妥当性が実地で再確認できた）。

```
レシートPDF(複数枚)
      │  POST /infer-receipt
      ▼
┌────────────────────── Hono Worker ──────────────────────┐
│ ① 抽出段: Gemini(マルチモーダル) + responseJsonSchema      │
│      tools 無し → レシート配列 [{vendor,date,lineItems}]   │
│                                                          │
│ ② 推論段(レシートごと): Gemini ── AFC ──┐                 │
│      tools:[mcpToTool(client)]          │ モデルが自分で呼ぶ│
│        ├─▶ query-index   (RAG: per-client Vectorize)     │
│        └─▶ validate-infer (型 + 有効勘定科目の検証)        │
│      └── 型エラーが消えるまでモデルが再呼び ──┘            │
│      失敗したら温度を上げて再試行（自前ループ・最大3回）    │
└──────────────────────────────────────────────────────────┘
      │
      ▼
  トレースJSON（レシートごとに: 抽出結果 / 呼ばれたツール /
   query-indexの過去仕訳とscore / 推論結果 / validate結果 /
   requiredHumanReview / 実測コスト）
```

推論段の MCP サーバ（`query-index` + `validate-infer`）は**同一 Worker 内**に置き、
モデルは `mcpToTool` 経由（InMemoryTransport）で接続する。【デモ】（本番は別 Worker に分離。理由は §3.2。）

## 3. 設計判断と理由

### 3.1 なぜ「モデル駆動（AFC）」で、一発の構造化出力にしなかったか

**当時のモデルは structured output を指定しても確実には守らなかった**（型安全な出力が難しかった）。【本番✓】
クレカ・通帳は半構造化済みなので `responseJsonSchema` の**一発出力**で足りたが、レシートはそれでは破綻した。

そこで発想を変え、**MCP の `validate-infer` に型エラー・不正な勘定科目を出させ、
モデルにエラーが消えるまで MCP を叩かせて確実に妥当な出力を作らせる**。【本番✓】
`validate-infer` は勘定科目のガードレールであると同時に、**スキーマ順守をツールループで強制する仕組み**でもある。

### 3.2 なぜ MCP か（手製のツールコールループにしなかった理由）

これがレシート経路を別構造にした決定打。手製のツールコールループは、Gemini API を何度も叩くため：【本番✓】

1. **CPU time が増大**する（Cloudflare Workers の課金対象）。
2. **1 呼び出しあたりのサブリクエスト（外部フェッチ）回数上限**に抵触し、1 Worker 内での実装は実質不可能だった。

そこで **MCP を別 Worker（service binding）に切り出し**、embedding / Vectorize / D1 のサブリクエストを
MCP 側 Worker の予算に逃がした。＝**プラットフォーム制約を理解した上での分割**。
（注：このサブリクエスト上限は当時の制約で、現在は緩和／撤廃されたとのこと。）

> 【デモ】簡略化のため MCP は**同一 Worker 内にマウント**する（動かしやすさ優先）。
> 上記の「別 Worker に分けた理由」は本デモでは再現せず、ここに記録するに留める。

### 3.3 なぜ RAG をモデルに自由に呼ばせるか（per-client namespace）

レシートは **1 発の検索では良い回答が得られない**ため、`query-index` をツール化し、
**モデルが必要なだけ自分で呼べる**ようにした。【本番✓】

検索は **クライアントごとの Vectorize namespace** に隔離する。【本番✓】
理由：**同じ店でもクライアントによって付ける勘定科目の癖が違う**ため、
そのクライアント自身の過去仕訳に根拠付ける必要がある（他クライアントの仕訳を引いてはいけない）。

- 埋め込み：`gemini-embedding-001` / 768 次元 / 取り込み時 `RETRIEVAL_DOCUMENT`・検索時 `RETRIEVAL_QUERY`。【本番✓・デモ✓（同一）】

### 3.4 リトライは 2 層ある（混同しない）

- **(a) 1 回の生成内（AFC 自己修正）**：`generateContent` 1 回の中で、モデルが `validate-infer` の
  エラーを見て**型エラーが消えるまで再呼び出し**する。**この層では温度は固定**。【本番✓】
- **(b) 試行（attempt）間（温度エスカレーション）**：(a) でも妥当な出力に至らず失敗した場合に再試行し、
  **試行ごとに temperature を 0.2 → 1.0 へ段階的に上げる**（topP/topK も上昇）。【本番✓】
  理由：**温度を固定（特に 0）にすると同じ誤答を延々繰り返す**。揺らぎを増やして詰まった誤答から脱出させる。

> 本番では (b) の試行カウントは **Cloudflare Queue の再配信（`message.attempts`）**に依存していた。【本番✓】
> 【デモ】Queue を削るため、(b) は**ハンドラ内の自前リトライループ**で再現する（attempt 1..5 で温度を上げて再生成）。
> キュー再配信の挙動はデモには無い。

### 3.5 requiredHumanReview（要確認フラグ）

アプリを実運用すると、**全件チェックの時間そのものが新たなボトルネック**になった。【本番✓】
そこで **モデルが「確認してほしい」ものだけにフラグ**を立て、人間はそこを重点的に見ればよくした
（STREAMED にも同等の要確認フラグ機能があり、機能パリティでもある）。【本番✓】
人間による最終確認は残しつつ、主眼は**レビュー工数の削減**。

### 3.6 レシート PDF の読み取り

【本番✓】顧客は**数千ページの単一 PDF** を送ってくることが多々あった。一方 **Gemini は 1 PDF を約 135〜150 ページ
しか読まず、超過分を黙って落とす**（現在も同様）。全ページを確実に処理するため、PDF を**ページ分割**し、
OCR/抽出を独立段（`extract_text` キュー）として回していた。

> 【デモ】入力を「**1 PDF・数枚のレシート**」に制約。**抽出段**（`responseJsonSchema`・tools 無し）で
> マルチモーダル直入力からレシート配列を取り出し、**推論段**でレシートごとに分類する（本番の
> `extract_text` → per-page `infer` と同じ 2 段）。PDF 分割・Queue・OCR 独立 Worker は省く。

### 3.7 モデル選択（抽出 flash / 推論 pro）と、本番との正直な差

- **抽出段＝`gemini-2.5-flash`**：`responseJsonSchema` の構造化出力で安定、かつ安い。【本番✓と同じ】
  （本番も `EXTRACT_TEXT_RECEIPT_MODEL = gemini-2.5-flash`。）
- **本番の推論段は `gemini-2.5-flash`**（`INFER_MODEL = gemini-2.5-flash`、queue は `model = env.INFER_MODEL`）。【本番✓】
  flash は `validate-infer` の関数呼び出しを確率的に `MALFORMED_FUNCTION_CALL` で破綻させる（実測 ~13%/試行）。
  本番はこれを **Cloudflare Queue の再配信（`message.attempts` ＝多数の非同期リトライ）＋温度エスカレーション**で
  吸収していた（§3.4 と整合）。
- **デモの推論段＝`gemini-2.5-pro`【デモ・意図的な妥協】**：デモは Queue を持たないため、本番のような多数回リトライで
  flash の破綻を擦り抜けるのは非現実的（~13%/試行では 6/6 を数回で揃えられない）。そこで**信頼性を取り、推論段だけ
  pro**にした（6/6 が 1 回目で成功）。＝「本番＝flash＋Queue 再配信」「デモ＝pro＋短いループ」で**同じ結果に別経路で到達**。
- 面接で「本番が flash なのに、なぜデモは pro か」と問われたら上記をそのまま答える（盛らない）。
  本番代表のコストは flash 基準で見るべき（§6）。

## 5. データ（すべて安全：合成 or 自分の匿名レシート）

1. **勘定科目表（合成 / D1）** — 日本の一般的な科目 10〜15 個（消耗品費・会議費・交際費・旅費交通費・通信費・
   新聞図書費・支払手数料・地代家賃・水道光熱費・雑費 等）。`is_active` を持たせ、**1 つを非アクティブ**にして
   「無効科目の拒否」も見せる。`validate-infer` の照合元。
2. **過去仕訳（合成 / Vectorize, namespace=`demo-client-001`）** — 10〜20 件。サンプルレシートと意図的に対応付ける：
   - 数件はサンプルと**似た店/文脈**（→ `query-index` が高 score でヒット＝RAG が効く画）
   - 1 件は**わざと過去に無い店**（→ low score / 該当なし → `requiredHumanReview` true の画）
3. **サンプルレシート（自分個人のもの・匿名化 / `sample/`）** — 3〜5 枚。**氏名・カード番号・住所はマスキング**。
   **顧客のレシートは使わない（厳守）**。少なくとも 1 枚は「過去に無い店」にしてフラグ発火を見せ、
   可能なら**存在しない/無効な勘定科目に誘導**して `validate-infer` の拒否＋リトライも 1 回見せる。

## 6. コスト

- **STREAMED：1 枚 ≒ ¥20**（事務所が実際に支払っていた額）。【本番✓】
- **make-account-item：Gemini API コストのみ／1 枚 ¥1 未満**（確実）。【本番✓】
- **本番代表値（flash 実測, 2026-06）≒ ¥0.13/枚**：抽出(flash) ＋ 推論(flash) で、デモ実行中に
  flash で完了したレシートの実測が 1 枚 ¥0.08〜0.11、抽出按分 ¥0.035 を加えて ≒ **¥0.13/枚**。
  本番もモデル構成は flash なので、**「1 枚 ¥1 未満／STREAMED の約 150 分の 1（≒2 桁安い）」の根拠はこの値**。
- **デモ実測（pro 推論, 2026-06）≒ ¥0.49/枚**：6 枚で合計 ¥2.94（抽出 flash 全体 ¥0.21 ＋ 推論 pro 各 ¥0.43〜0.49）。
  デモが信頼性のため推論を pro にしたぶん高いが、**それでも 1 枚 ¥1 未満**。トレース JSON の `totalCostYen` / `costYen` に出る。
- precise な極小値は断定しない（モデル進化で変動）。本番主張は flash 基準（¥0.13）、デモの数字（¥0.49）は pro 固有と区別する。

## 7. 使い方（予定）

```bash
pnpm dev                 # Hono Worker 起動（MCP マウント込み）
pnpm db:schema && pnpm db:seed     # 合成勘定科目を D1 へ
pnpm seed:vectors        # 合成過去仕訳を Vectorize へ（demo-client-001）
curl -F file=@sample/receipt-1.pdf http://localhost:8787/infer-receipt
```

`curl` のレスポンス（トレース JSON）は **レシートごとの配列** `results[]`：各要素に抽出されたレシート・
`trace`（モデルが呼んだツールの時系列＝`query-index` の過去仕訳と score、`validate-infer` の成否）・
推論結果（`inferred.data` / `requiredHumanReview`）・実測コスト・`finishReason` を載せ、JSON で「物語」を見せる。
トップに `ok`（全レシート成功か）・`receiptsCount`・`totalCostYen`。

## 8. 限界・今後

- これは **agent コアの忠実な抽出**であり、本番のスケール用インフラ（Queue/Workflow/DLQ/分割）は持たない（§4）。
- 精度は RAG の過去仕訳の質に強く依存する（クライアント固有の癖を学ぶほど良くなる）。
- requiredHumanReview は**過検知寄り**に倒すのが安全（見落としより、人手確認が増える方がまし）。
